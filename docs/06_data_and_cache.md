# 06. Данные, кэш и архивный слой

## Общая модель хранения

В архитектуре используются три уровня хранения:

- транзакционный слой на `PostgreSQL`;
- горячий слой на `Redis / Valkey`;
- архивный слой `NoSQL / cold storage`.

Такое разделение позволяет не смешивать подтвержденные бизнес-данные, быстрые read-модели и длительное хранение истории.

## PostgreSQL как source of truth

`PostgreSQL` остается основным source of truth для транзакционных и справочных данных. При этом он трактуется не как одна полностью общая БД на все сервисы, а как единая технологическая платформа с logical ownership boundaries по сервисам. В учебной версии это можно описывать как:

- отдельные схемы;
- или отдельные базы в одном кластере `PostgreSQL`.

### Что хранится в PostgreSQL

`PostgreSQL` хранит:

- активные записи очереди;
- активные бронирования;
- статусы зон;
- лимиты и правила зон;
- окна доступности;
- прикладной профиль пользователя;
- административные изменения;
- outbox-записи;
- idempotency-записи;
- журналы уведомлений;
- raw payload событий через `jsonb`.

### Почему PostgreSQL здесь оправдан

Именно этот слой подтверждает критичные транзакции. Поэтому здесь должны оставаться данные, от которых зависит окончательный ответ пользователю или администратору.

## Redis / Valkey

`Redis / Valkey` используется как горячий слой для read-моделей, технических ключей и механизмов защиты входного контура.

### Что хранится в Redis / Valkey

Типовые ключи и данные:

- `zone:status:{zoneId}`
- `zone:summary:{zoneId}`
- `queue:summary:{zoneId}`
- `queue:position:{zoneId}:{userId}`
- `reservation:availability:{zoneId}:{slotStart}`
- `user:access:{userId}`
- `ratelimit:{route}:{principal}`
- `idempotency:{operation}:{requestId}`
- lock keys
- краткоживущие technical keys

### Зачем нужен этот слой

`Redis / Valkey` нужен, чтобы:

- снять read-нагрузку с `PostgreSQL`;
- ускорить UI и частые запросы чтения;
- хранить временные счетчики и данные rate limiting;
- держать краткоживущие idempotency window keys;
- использовать технические lock keys там, где нужен короткий защитный механизм.

## TTL и invalidation

### Рекомендуемые диапазоны TTL

- zone, queue и reservation read-модели: короткий TTL `3-30 сек`;
- `user:access:{userId}`: `5-15 мин`;
- idempotency keys: от `15 мин` до `24 ч` в зависимости от операции;
- rate limit keys: по окну выбранного лимита.

### Как обновляется кэш

Для read-моделей используется комбинированный подход:

- короткий TTL;
- инвалидирование по доменным событиям.

Это важно, потому что только TTL без событийного обновления дает лишний лаг, а только событийная инвалидция делает систему слишком чувствительной к пропускам в асинхронном контуре.

## NoSQL / cold storage

В реализованном MVP `NoSQL / cold storage` конкретизировано через `MongoDB`. Отдельное `S3-compatible object storage` в текущую локальную версию не включено.

### Что хранится в MongoDB

- `occupancy_events` с raw payload и publish status;
- `telemetry_snapshots` с последним наблюдением по зоне;
- история occupancy-сигналов для demo и ручного анализа.

### Что остается в оперативном контуре PostgreSQL

В `PostgreSQL` сохраняются:

- транзакционные данные;
- журналы, которые еще нужны в операционной работе;
- outbox;
- idempotency;
- raw payload там, где они нужны producer-сервисам для outbox и журналирования.

## Риски и компромиссы кэширования

### Stale data

Горячие read-модели могут отставать от транзакционного слоя. Для этой системы это особенно чувствительно в сценариях очереди и доступности мест.

### Сложность invalidation

Чем больше производных ключей и read-моделей, тем проще ошибиться в их обновлении. Это создает риск того, что разные экраны будут видеть немного разное состояние одной `managed zone`.

### Cache stampede

При коротких TTL и пиковом интересе к одной зоне есть риск массового одновременного пересоздания кэша. В учебном проекте достаточно зафиксировать сам риск и не углубляться в сложные антиштормовые техники.

## Консистентность по слоям

### Транзакционный слой

`PostgreSQL` фиксирует подтвержденный результат команд и остается опорным слоем для критичных операций.

### Горячий слой

`Redis / Valkey` может кратковременно отставать. Это допустимо для read-моделей, счетчиков и лимитов, если транзакционный ответ пользователю формируется из подтвержденного состояния.

### Архивный слой

`MongoDB` наполняется отдельно и не участвует в подтверждении пользовательской операции. Поэтому здесь допустим больший лаг и фоновая републикация для failed occupancy events.

## Практический вывод

В этой архитектуре:

- `PostgreSQL` отвечает за подтвержденные бизнес-данные, журналы, outbox и idempotency;
- `Redis / Valkey` отвечает за hot read-модели, счетчики, лимиты и краткоживущие технические ключи;
- `MongoDB` отвечает за history и telemetry по occupancy signals.

Такой подход остается достаточно простым для учебного high-level проекта, но при этом честно показывает компромисс между latency, консистентностью и эксплуатационной сложностью.
