# 04. Обзор архитектуры

## Архитектурная идея

Система строится вокруг общей модели `managed zone / shared university space`. Для пользователя это единый контур управления состоянием зоны, очередью, бронированием и уведомлениями. Для архитектуры это означает разделение на:

- синхронный слой пользовательских и административных команд;
- асинхронный слой обработки событий и вторичных реакций;
- слой хранения данных с разными режимами работы: транзакционные данные, горячий кэш и архивная история.

## Основные элементы

### Клиенты

- `User Client App` - пользовательский интерфейс для студентов и сотрудников;
- `Admin Web App` - административный интерфейс для управления зонами и правилами.

### Routing layer

- `Edge Router / Load Balancer` - входной сетевой слой;
- `API Gateway` - единая прикладная точка входа в сервисы, где также применяются политики доступа и rate limiting.

### Сервисы

- `User Service` - сведения о пользователях, ролях и проверках доступа;
- `Zone Management Service` - текущее состояние `managed zone`, статус зоны, лимиты и правила;
- `Queue Service` - операции очереди и связанное состояние;
- `Reservation Service` - операции бронирования и связанное состояние;
- `Notification Service` - отправка уведомлений и журналы доставки;
- `Monitoring / Event Ingestion Service` - прием внешних сигналов о загрузке зоны и их публикация в событийный поток;
- `Observability Stack` - техническая наблюдаемость, метрики, логи и трассировка.

## Принцип взаимодействия

### Синхронный контур

Синхронный контур нужен для операций, где пользователю нужен немедленный результат:

- чтение состояния зоны;
- получение статуса очереди;
- просмотр доступности мест;
- вступление в очередь и выход из очереди;
- создание и отмена бронирования;
- административное изменение статуса зоны, лимитов и правил.

Для этих операций запрос проходит через `Edge Router / Load Balancer` и `API Gateway`, затем попадает в нужный доменный сервис. При необходимости сервис синхронно запрашивает проверки ролей и правил зоны у `User Service` и `Zone Management Service`.

### Асинхронный контур

Асинхронный контур нужен для событий и вторичных реакций:

- обработка внешних сигналов загрузки;
- публикация доменных событий через Kafka;
- пересчет агрегированного состояния зоны;
- реакция на перегрузку;
- уведомления пользователям;
- накопление истории событий и архивных данных.

Основная идея состоит в том, что подтверждение бизнес-команды не зависит от завершения вторичных действий. После фиксации результата сервис публикует событие в Kafka, а остальные реакции выполняются отдельно.

## Хранилища и их место

### PostgreSQL

`PostgreSQL` используется как единая технологическая платформа для транзакционных данных. При этом данные не описываются как полностью общая БД для всех сервисов: в учебной версии предполагается логическое разделение по сервисам, например через отдельные схемы или отдельные базы в одном кластере.

### Redis / Valkey

`Redis / Valkey` хранит горячие данные, счетчики, кэш состояния зоны и данные для rate limiting. Это снижает нагрузку на транзакционное хранилище и помогает быстрее отдавать часто запрашиваемое состояние.

### NoSQL / cold storage

В реализованном MVP этот слой конкретизирован через `MongoDB`. Он используется для истории событий загрузки зон, журналов `occupancy_updated` и telemetry snapshots. Слой не участвует в критическом транзакционном пути, но нужен для ретроспективного анализа и демонстрации внешних сигналов загрузки.

## Почему архитектура не делается сложнее

В проекте сознательно не вводятся дополнительные сервисы оркестрации, отдельные выделенные аналитические контуры и детальные схемы интеграций. Для учебной high-level архитектуры достаточно показать:

- где подтверждаются ключевые команды;
- где начинается асинхронная обработка;
- какие данные считаются транзакционными, горячими и архивными;
- как routing layer защищает систему от перегрузки.
