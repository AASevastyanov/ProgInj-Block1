# 03. User Stories, Use Cases и NFR

## Пользовательские истории

### Студент

- Как студент, я хочу видеть текущее состояние `managed zone`, чтобы понимать, стоит ли идти в столовую или искать место в коворкинге.
- Как студент, я хочу вступить в очередь и выйти из очереди, чтобы управлять своим посещением зоны со сценариями ожидания.
- Как студент, я хочу создавать и отменять бронирование, чтобы планировать использование мест в зоне со сценариями резервирования.
- Как студент, я хочу получать уведомления об изменении состояния зоны, чтобы не отслеживать его вручную.

### Преподаватель / сотрудник университета

- Как преподаватель / сотрудник университета, я хочу использовать те же сценарии просмотра состояния зоны, очереди и бронирования, чтобы планировать посещение общих университетских пространств.
- Как преподаватель / сотрудник университета, я хочу, чтобы правила зоны учитывали мою роль, если для нее предусмотрены отдельные лимиты или приоритеты.

### Администратор столовой

- Как администратор столовой, я хочу менять статус зоны, лимиты и правила, чтобы управлять потоком посетителей в режиме реального времени.
- Как администратор столовой, я хочу видеть признаки перегрузки зоны, чтобы своевременно вводить ограничения.

### Администратор коворкинга

- Как администратор коворкинга, я хочу управлять статусом зоны, доступностью мест и правилами бронирования, чтобы контролировать использование рабочих мест.
- Как администратор коворкинга, я хочу получать информацию о перегрузке и изменении состояния зоны, чтобы корректировать правила доступа.

### Системный администратор

- Как системный администратор, я хочу видеть техническое состояние системы и поток обработки событий, чтобы поддерживать стабильность сервиса.

## Ключевые use cases

### UC-1. Просмотр состояния зоны

Пользователь через `User Client App` запрашивает текущее состояние `managed zone`. Система синхронно возвращает актуальное агрегированное состояние, статус зоны и, если применимо, краткий статус очереди или доступности мест.

### UC-2. Получение статуса очереди

Пользователь запрашивает текущее состояние очереди для выбранной зоны. Система синхронно возвращает позицию, длину очереди или признак недоступности очереди для данной зоны.

### UC-3. Вступление в очередь

Пользователь инициирует команду вступления в очередь. Система синхронно проверяет роль пользователя, правила зоны и текущие ограничения, фиксирует результат и после подтверждения публикует доменное событие.

### UC-4. Выход из очереди

Пользователь инициирует выход из очереди. Система синхронно подтверждает изменение, а вторичные реакции выполняются асинхронно.

### UC-5. Просмотр доступности мест

Пользователь запрашивает доступность мест в `managed zone`. Система возвращает состояние на основе актуальных данных и правил зоны.

### UC-6. Создание бронирования

Пользователь создает бронирование места. Система синхронно проверяет права, лимиты и доступность, подтверждает результат и публикует событие `reservation_created`.

### UC-7. Отмена бронирования

Пользователь отменяет ранее созданное бронирование. Система синхронно фиксирует отмену и публикует событие `reservation_cancelled`.

### UC-8. Изменение правил зоны администратором

Администратор через `Admin Web App` меняет статус зоны, лимиты или правила доступа. Система подтверждает изменение синхронно, а связанные вторичные реакции распространяются асинхронно.

### UC-9. Асинхронная реакция на изменение загрузки зоны

`Monitoring / Event Ingestion Service` принимает внешний сигнал загрузки зоны, публикует `occupancy_updated`, после чего другие сервисы пересчитывают агрегированное состояние, фиксируют перегрузку и рассылают уведомления.

## Нефункциональные требования

### Производительность

- чтение состояния зоны и доступности мест должно оставаться быстрым даже при пиковом числе обращений;
- ключевые команды пользователя не должны зависеть от завершения вторичных асинхронных действий;
- для горячих данных и rate limiting используется `Redis / Valkey`.

### Консистентность

- транзакционный результат основной команды должен подтверждаться синхронно;
- вторичные реакции после подтверждения команды допускают eventual consistency;
- агрегированное состояние зоны после внешнего `occupancy_updated` может обновляться с небольшим лагом.

### Надежность

- отказ одного асинхронного потребителя не должен блокировать прием основных пользовательских команд;
- события должны быть доступны для повторной обработки в рамках принятой event-driven модели;
- история загрузки и архивные журналы не должны зависеть только от горячего кэша.

### Безопасность и контроль доступа

- проверки роли пользователя и правил зоны обязательны в пользовательских и административных операциях;
- административные команды должны иметь более строгие лимиты и контроль доступа, чем пользовательские запросы чтения;
- routing layer должен ограничивать злоупотребление API.

### Масштабируемость и эксплуатация

- рост числа запросов на чтение и числа асинхронных событий должен обрабатываться независимо;
- архитектура должна позволять отдельно масштабировать API-слой, сервисы обработки команд и обработчики событий;
- технические метрики и журналы должны собираться в `Observability Stack`.
